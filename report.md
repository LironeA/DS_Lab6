# Лабораторна робота №6

**Дисципліна:** Розподілені системи

## 1. Мета роботи

Метою роботи є моделювання тристороннього `Atomic Swap` між трьома незалежними процесами з використанням навчальної реалізації `HTLC (Hashed Timelock Contract)`, перевірки секрету, таймлоків, операцій `redeem` і `refund`, а також журналювання всіх ключових подій.

## 2. Постановка задачі

Потрібно реалізувати обмін умовними криптоактивами між `Party A`, `Party B` і `Party C`.  
Кожна сторона має бути окремим процесом. Взаємодія між процесами виконується через JSON-файли у спільному каталозі.  
Програма повинна підтримувати:

- успішний сценарій тристороннього atomic swap;
- сценарій з таймаутом і поверненням активів через `refund`;
- логування всіх етапів роботи;
- показ фінальних балансів і фінальних станів контрактів.

## 3. Теоретичні відомості

### 3.1. Що таке Atomic Swap

`Atomic Swap` це механізм обміну активами між учасниками без необхідності довіряти один одному. Обмін називається атомарним, тому що він або виконується повністю, або не виконується зовсім.

### 3.2. Що таке HTLC

`HTLC (Hashed Timelock Contract)` це контракт з двома умовами:

- `Hash Lock`: для отримання активу потрібно надати правильний секрет `S`, для якого заздалегідь відомий `hash(S)`.
- `Time Lock`: якщо секрет не розкрито до певного моменту часу, відправник може повернути актив собі.

### 3.3. Що таке hash lock

`Hash lock` це умова перевірки секрету. У контракті зберігається не сам секрет, а лише його хеш. Якщо під час `redeem` хеш наданого секрету співпадає зі значенням у контракті, отримувач може забрати актив.

### 3.4. Що таке time lock

`Time lock` це обмеження по часу. Воно не дозволяє контракту залишатися заблокованим безкінечно. Після завершення таймлоку відправник може виконати `refund`.

### 3.5. Навіщо потрібен refund

`Refund` гарантує, що у випадку невдалого обміну активи не будуть втрачені. Якщо ланцюжок `redeem` не завершився, кожен учасник після закінчення таймлоку повертає свій заблокований актив.

### 3.6. Чому тристоронній обмін складніший за двосторонній

У тристоронньому обміні секрет і порядок дій повинні передаватися по ланцюгу `A -> C -> B`. Якщо хоча б одна ланка не завершує потрібну дію вчасно, весь обмін порушується. Через це важливо правильно узгодити таймлоки і забезпечити можливість повернення коштів.

## 4. Опис реалізованої моделі

### 4.1. Сторони A, B, C

У системі є три сторони:

- `PartyA` володіє `CoinA`;
- `PartyB` володіє `CoinB`;
- `PartyC` володіє `CoinC`.

Початкові баланси:

- `PartyA`: `CoinA = 100`, `CoinB = 0`, `CoinC = 0`
- `PartyB`: `CoinA = 0`, `CoinB = 100`, `CoinC = 0`
- `PartyC`: `CoinA = 0`, `CoinB = 0`, `CoinC = 100`

### 4.2. Активи і контракти

Успішний ланцюг обміну:

- `A -> B`: `10 CoinA`
- `B -> C`: `15 CoinB`
- `C -> A`: `20 CoinC`

Кожен контракт має поля:

- `ContractId`
- `FromParty`
- `ToParty`
- `AssetName`
- `Amount`
- `HashLock`
- `TimeLockUtc`
- `Status`

Також зберігаються службові поля часу фінансування, викупу, повернення та розкритий секрет.

### 4.3. Процеси

У програмі працюють окремі процеси:

- `Coordinator`
- `PartyA`
- `PartyB`
- `PartyC`

`Coordinator` створює каталог запуску, зберігає конфігурацію сценарію, запускає три дочірні процеси і після завершення формує підсумок.

### 4.4. Обмін через файли / JSON

Стан зберігається у каталозі `runs/<scenario>/state/`:

- `scenario.json`
- `balances.json`
- `contract_ab.json`
- `contract_bc.json`
- `contract_ca.json`

Процеси читають і змінюють ці файли незалежно. Для простого захисту від конфліктів запису використовується файловий lock.

### 4.5. Логування

Кожен процес пише окремий лог у `runs/<scenario>/logs/`.  
Формат повідомлення:

```text
[timestamp] [process] [level] message
```

## 5. Опис роботи програми

### 5.1. Меню

Застосунок має меню:

1. запуск успішного сценарію;
2. запуск timeout/refund сценарію;
3. показ фінальних логів останнього запуску;
4. вихід.

### 5.2. Успішний сценарій

1. `PartyA` генерує секрет `S` і обчислює `SHA256(S)`.
2. `PartyA` створює і фінансує контракт `A -> B`.
3. `PartyB` читає `hash lock`, створює і фінансує контракт `B -> C`.
4. `PartyC` створює і фінансує контракт `C -> A`.
5. `PartyA` робить `redeem` у контракті `C -> A` і розкриває секрет.
6. `PartyC` використовує розкритий секрет для `redeem` у контракті `B -> C`.
7. `PartyB` використовує той самий секрет для `redeem` у контракті `A -> B`.
8. Усі контракти завершуються зі статусом `Redeemed`.

### 5.3. Timeout-сценарій

1. Контракти створюються і фінансуються аналогічно.
2. `PartyA` навмисно не виконує `redeem` у контракті `C -> A`.
3. Секрет не розкривається.
4. Після завершення таймлоків сторони виконують `refund` для своїх контрактів.
5. Контракти переходять у статус `Refunded`.

### 5.4. Фінальні стани

Програма після завершення сценарію виводить:

- список контрактів і їх фінальні стани;
- фінальні баланси всіх сторін;
- шлях до логів конкретного запуску.

## 6. Як працює HTLC у цій реалізації

У кожному контракті зберігається `HashLock`, який однаковий для всіх трьох HTLC.  
Операція `Redeem(secret)` перевіряє:

- чи контракт у статусі `Funded`;
- чи дію виконує правильний отримувач;
- чи не завершився таймлок;
- чи збігається `SHA256(secret)` із `HashLock`.

Якщо умови виконано, контракт переходить у `Redeemed`, а секрет зберігається у полі `RevealedSecret`. Це дозволяє іншим сторонам прочитати секрет із JSON-файлу і завершити свої `redeem`.

## 7. Як відбувається розкриття секрету

Секрет генерує `PartyA`. Спочатку інші сторони знають тільки хеш.  
Під час успішного сценарію `PartyA` виконує `redeem` для контракту `C -> A` і записує секрет у стан контракту.  
Після цього:

- `PartyC` читає секрет і викуповує `B -> C`;
- `PartyB` читає той самий секрет і викуповує `A -> B`.

Таким чином один і той самий секрет проходить по ланцюгу та завершує весь обмін.

## 8. Як працює refund

Якщо таймлок закінчився, а `redeem` не відбувся, відправник виконує `Refund()`.  
У цьому методі перевіряється:

- чи контракт ще не був `Redeemed`;
- чи виклик робить саме відправник;
- чи таймлок уже завершився.

Після успішного `refund` кошти повертаються на баланс початкового власника, а контракт отримує статус `Refunded`.

## 9. Як координуються окремі процеси

Координація реалізована максимально просто:

- `Coordinator` запускає процеси через аргументи командного рядка;
- стан сценарію зберігається у спільному каталозі;
- сторони очікують появу або зміну JSON-файлів через polling з коротким `Thread.Sleep`;
- для запису використовується файлове блокування.

Це дозволяє показати принцип розподіленої взаємодії без складного IPC.

## 10. Чому це atomic swap у навчальному сенсі

Успішний сценарій показує, що всі учасники отримують потрібні активи лише після розкриття секрету.  
Timeout-сценарій показує, що без розкриття секрету кошти не губляться і повертаються власникам.  
Саме ця властивість "успіх для всіх або повернення всім" і відтворює ідею atomic swap у спрощеній навчальній моделі.

## 11. Результати роботи

### 11.1. Приклад успішного запуску

Очікуваний результат:

- усі три контракти мають статус `Redeemed`;
- баланси стають такими:
  `PartyA`: отримує `20 CoinC`
  `PartyB`: отримує `10 CoinA`
  `PartyC`: отримує `15 CoinB`

### 11.2. Приклад timeout-запуску

Очікуваний результат:

- секрет не розкривається;
- усі контракти завершуються через `Refunded`;
- баланси повертаються до початкових значень.

### 11.3. Короткий аналіз логів

За логами можна простежити:

- хто створив контракт;
- хто його профінансував;
- коли було розкрито секрет;
- як інші сторони використали той самий секрет;
- у якому місці сценарій зупинився при timeout;
- які `refund` були виконані після завершення таймлоків.
